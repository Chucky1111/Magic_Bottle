# 用户灵感注入功能开发计划

## 1. 项目概述
本项目旨在为小说写作系统添加"用户灵感注入"功能，允许用户管理自己的创作灵感，并在设计（design）和写作（write）阶段自动注入这些灵感，以提升创作质量与个性化程度。

## 2. 需求分析

### 2.1 核心需求
1. **灵感文件管理**：在 `prompts/user_idea/` 目录下存储灵感文件（`.txt`格式）
2. **文件格式**：文件内容用 `---` 分隔不同灵感条目，已完成条目前标记 `[已完成]`
3. **注入机制**：
   - 遍历 `prompts/user_idea/` 目录下所有 `.txt` 文件
   - 排除文件名已有 `completed_` 前缀的文件
   - 读取文件中未标记 `[已完成]` 的灵感条目
   - 将灵感条目注入到 `prompts/design.txt` 和 `prompts/write.txt` 的 `{{user_idea}}` 占位符
   - 每个未完成条目在design和write阶段各注入一次
4. **状态更新**：
   - 注入完成后，在条目前添加 `[已完成]` 标记
   - 当文件内所有条目均标记为完成，将文件名加上 `completed_` 前缀
5. **断点续传**：系统重启后能从上次中断位置继续处理

### 2.2 技术约束
- 遵循项目"高内聚、低耦合"架构原则
- 所有外部I/O操作需包含重试机制（使用 `tenacity` 库）
- 状态持久化：每一步操作后立即写入硬盘（JSON格式）
- 模块化设计：严禁将所有代码堆在一个文件中

### 2.3 非功能性需求
- **健壮性**：处理文件不存在、格式错误等异常情况
- **性能**：支持大量灵感条目的高效处理
- **可维护性**：代码清晰，便于后续扩展

## 4. 技术设计方案

### 4.1 架构设计
采用分层设计，新增以下组件：

1. **UserIdeaManager** (`services/user_idea_manager.py`)
   - 负责灵感文件的读取、解析、更新
   - 维护注入状态，支持断点续传
   - 提供获取未完成灵感条目的接口

2. **Enhanced PromptRenderer** (`services/prompt_renderer.py`)
   - 集成 `UserIdeaManager`
   - 在渲染design和write提示词时注入灵感内容

3. **State Extension** (`models/state.py`)
   - 扩展状态模型，记录灵感注入进度

4. **Configuration** (`config/settings.py`)
   - 添加用户灵感相关配置项

### 4.2 类图设计
```
┌─────────────────┐      ┌──────────────────┐      ┌──────────────────┐
│  PromptRenderer │      │ UserIdeaManager  │      │   StateManager   │
├─────────────────┤      ├──────────────────┤      ├──────────────────┤
│ +render_design()│─────>│ +get_ideas()     │<─────│ +get_injection() │
│ +render_write() │      │ +mark_completed()│      │ +set_injection() │
└─────────────────┘      │ +rename_file()   │      └──────────────────┘
                         └──────────────────┘
```

### 4.3 数据流设计
```
1. 工作流启动
   ├─ 2. PromptRenderer.render_design_prompt()
   │   ├─ 3. 调用 UserIdeaManager.get_next_idea()
   │   ├─ 4. 获取未完成灵感条目
   │   ├─ 5. 替换 {{user_idea}} 占位符
   │   └─ 6. 返回注入后的提示词
   │
   ├─ 7. LLM生成设计内容
   │
   ├─ 8. PromptRenderer.render_write_prompt()
   │   ├─ 9. 调用 UserIdeaManager.get_same_idea()
   │   ├─ 10. 获取相同的灵感条目
   │   ├─ 11. 替换 {{user_idea}} 占位符
   │   └─ 12. 返回注入后的提示词
   │
   └─ 13. LLM生成写作内容
        │
        └─ 14. 灵感条目标记为已完成
             │
             └─ 15. 检查文件是否全部完成，如是则重命名
```

### 4.4 异常处理设计
1. **文件不存在**：优雅降级，返回空字符串
2. **格式错误**：记录警告日志，跳过错误条目
3. **重试机制**：文件操作使用 `tenacity` 库实现自动重试
4. **并发安全**：使用文件锁防止多进程同时修改

## 6. 风险评估与缓解措施

### 6.1 技术风险
| 风险 | 影响程度 | 概率 | 缓解措施 |
|------|----------|------|----------|
| 并发文件访问冲突 | 高 | 中 | 实现文件锁机制，使用 `utils/file_lock.py` |
| 状态不一致 | 高 | 低 | 定期状态一致性检查，添加回滚机制 |
| 性能瓶颈 | 低 | 低 | 灵感文件数量有限，性能影响可忽略 |

### 6.2 兼容性风险
- **占位符拼写问题**：已纠正为 `{{user_idea}}`，无需兼容旧拼写
- **旧版本兼容**：确保不影响现有工作流，无灵感文件时正常降级
