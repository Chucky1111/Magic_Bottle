# 定点重启模块开发规划

## 背景
小说生成系统是一个长周期运行的系统，每章包含design、write、summary三个阶段。系统状态保存在`data/state.json`中，历史对话记录保存在`data/history.json`中。为了应对意外中断（如程序崩溃、手动停止）以及支持“从某一章某一阶段重新开始”的需求，系统已实现快照机制（`data/history_snapes/`目录下自动生成的快照文件）。

然而，当前缺少一个**定点重启**工具，让用户能够便捷地选择某个历史快照，将系统状态和历史记录恢复到该快照对应的位置，并可以灵活调整章节和阶段。

## 需求分析
1. **核心需求**：用户可以从快照目录中选择一个快照文件，将系统恢复到该快照记录的状态。
2. **状态重置**：需要正确计算并更新`state.json`中的以下字段：
   - `chapter_num`：目标章节（可手动指定或使用快照记录的章节）
   - `stage`：目标阶段（design/write/summary）
   - `window_chapters`：根据目标章节和窗口大小动态计算
   - 其他字段（如`sequence_state`、`feedback_context`、`summary_status_counter`等）应沿用快照中保存的值，或根据新章节重新计算（若涉及跨章依赖）
3. **历史记录恢复**：将快照中的`history`字段完整复制到`data/history.json`，确保历史记录与状态一致。
4. **安全保护**：恢复前自动备份当前状态和历史记录，提供`--dry-run`预览模式。
5. **灵活性**：允许用户覆盖目标章节和阶段（例如从快照的chapter=3, stage=summary恢复到chapter=5, stage=write）。
6. **交互便利**：提供命令行参数直接指定快照，未来可扩展为交互式选择菜单。

## 设计思路
### 1. 模块定位
定点重启模块作为独立的工具脚本，不侵入主工作流。它读取快照文件（JSON格式），解析其中的`state_info`和`history`，然后按照用户指定的目标章节和阶段，更新`data/state.json`和`data/history.json`。

### 2. 快照格式兼容性
快照文件由`services/snapshot_manager.py`自动生成，格式如下：
```json
{
  "timestamp": "...",
  "chapter_range": [1,2,3],
  "state_info": { ... },  // 快照时刻的state.json完整内容
  "history": [ ... ]      // 快照时刻的history.json完整内容
}
```
模块必须确保能够解析该格式，并从中提取必要信息。

### 3. 状态计算逻辑
- **窗口章节计算**：根据目标章节`chapter_num`和`window_size`（默认为3）计算出`window_chapters`。例如`chapter_num=5, window_size=3` => `[3,4,5]`（若章节数不足则从1开始）。
- **阶段合法性校验**：目标阶段必须是`design`、`write`、`summary`之一。
- **序列状态调整**：如果快照中的`sequence_state`与目标章节不匹配（例如`design_index`、`write_index`），则按需重置（当前实现保留快照值，因为快照已包含对应章节的状态）。
- **其他字段**：直接使用快照中的`state_info`，但覆盖`chapter_num`、`stage`、`window_chapters`字段。

### 4. 历史记录恢复
- 将快照的`history`字段完整写回`data/history.json`。
- 注意：历史记录中的每条消息都带有`chapter`和`stage`标记，这些标记与快照时刻一致；恢复后，系统会根据当前状态过滤上下文，因此无需修改历史记录中的标记。

### 5. 备份与回滚
- 执行恢复前，自动将当前`state.json`和`history.json`复制为`state.json.bak`和`history.json.bak`。
- 若恢复过程中发生错误，提供回滚选项（目前为手动，可通过备份文件恢复）。

- 命令行参数解析（`--snapshot`、`--chapter`、`--stage`、`--no-history`、`--dry-run`）
- 快照加载与验证
- 状态计算与更新
- 历史记录恢复
- 备份创建
- 预览模式（dry-run）

## 使用方法
### 基本命令
```bash
# 恢复到指定快照（使用快照记录的章节和阶段）
python restore_snapshot.py --snapshot data/history_snapes/snapshot_20260124_110812_ch1-2-3.json

# 恢复到指定快照，并覆盖章节和阶段
python restore_snapshot.py --snapshot data/history_snapes/snapshot_20260124_110812_ch1-2-3.json --chapter 5 --stage write

# 仅恢复状态，不恢复历史记录（适用于仅重置进度）
python restore_snapshot.py --snapshot ... --no-history

# 预览恢复操作（不实际修改文件）
python restore_snapshot.py --snapshot ... --dry-run
```

### 交互式选择（未来扩展）
计划增加`--interactive`参数，列出所有快照供用户选择，并提示输入目标章节和阶段。

## 实现细节
### 关键代码片段
```python
def compute_window_chapters(target_chapter, window_size=3):
    """计算窗口章节列表"""
    start = max(1, target_chapter - window_size + 1)
    return list(range(start, target_chapter + 1))
```

### 错误处理
- 快照文件不存在或格式错误 → 抛出异常并退出
- 目标章节小于1 → 调整为1
- 目标阶段不在允许列表中 → 报错
- 文件读写权限问题 → 捕获IOError

## 测试方案
1. **单元测试**：针对状态计算函数编写测试用例。
2. **集成测试**：
   - 创建模拟快照文件，执行恢复操作，验证状态文件是否正确更新。
   - 恢复后运行主流程，确保系统能基于新状态继续生成。
3. **回归测试**：确保恢复操作不会破坏现有数据一致性。

## 未来改进方向
1. **交互式命令行菜单**：使用`curses`或`questionary`库提供更友好的选择界面。
2. **批量恢复**：支持一次恢复多个快照（合并历史记录）。
3. **快照过滤**：按章节范围、时间范围筛选快照。
4. **自动状态修复**：当快照中缺少某些字段时，自动从当前状态补充。
5. **集成到管理面板**：若将来有Web管理界面，可提供可视化恢复功能。

## 风险与注意事项
1. **数据一致性**：恢复操作会覆盖当前状态和历史记录，务必先备份。
2. **跨版本兼容**：如果快照格式随系统升级发生变化，需要版本适配逻辑。
3. **窗口章节计算**：若目标章节小于窗口大小，窗口将从第1章开始，可能导致历史上下文不足（需依赖系统的上下文修剪机制）。
4. **快照频率**：目前快照在每章完成后自动生成，若需要更细粒度的恢复点，可考虑增加阶段完成时的快照。

## 总结
定点重启模块为用户提供了灵活的状态恢复能力，是保证系统可靠性和可维护性的关键组件。当前已实现核心恢复功能，并通过命令行参数满足基本使用需求。未来可根据用户反馈逐步完善交互体验和高级功能。
