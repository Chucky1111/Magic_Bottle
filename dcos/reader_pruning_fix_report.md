# Reader历史记录修剪修复报告

## 问题分析
通过分析日志和代码，发现reader历史记录无限增长的问题：

1. **问题现象**：reader历史记录包含了所有已处理章节的消息（第1-10章），每个章节有2条消息（memerry + feedback）
2. **当前状态**：33条消息，其中13条基准上下文，20条章节消息
3. **根本原因**：reader工作流缺少修剪逻辑，导致历史记录随章节数线性增长

## 修复方案
在`services/reader_workflow.py`中添加了修剪逻辑：

### 核心修复代码（第505-586行）
```python
def _execute_pruning(self):
    """
    执行读者历史记录修剪逻辑
    
    当窗口满时执行反馈阶段后调用：
    1. 删除窗口中前两章的所有记录，保留第三章
    2. 保留基准上下文（系统提示词 + 预热对话）
    3. 保留最新的反馈消息（如果有）
    """
```

### 修剪策略
1. **保留基准上下文**：系统提示词 + 预热对话（前13条消息）
2. **保留最近3章**：窗口章节中的第三章（最新的一章）
3. **删除前两章**：窗口章节中的第一章和第二章
4. **删除其他旧章节**：确保历史记录只包含基准上下文 + 最近3章

- **窗口大小**：3章
- **处理流程**：积累3章 → 生成反馈 → 执行修剪 → 清空窗口
